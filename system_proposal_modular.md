# 쟈비스 프로젝트: LLM 한계 극복을 위한 모듈형 시스템 아키텍처 제안서

사용자님의 요청에 따라 LLM 기반 시스템의 고질적인 한계(환각, 불안정성, 문맥 길이 제한 등)를 극복하고, 유지보수 및 확장성이 용이한 **모듈형 에이전트 아키텍처(Modular Agent Architecture)**를 제안합니다. 각 모듈은 독립적으로 기능하면서도 유기적으로 연동되어 전체 시스템의 안정성과 효율성을 극대화합니다.

## 1. LLM 한계 극복 전략

LLM의 내재적 한계를 극복하기 위해 다음과 같은 전략을 아키텍처에 반영합니다.

| 한계점 | 극복 전략 | 적용 모듈 및 기술 |
| :--- | :--- | :--- |
| **환각(Hallucination) 및 부정확성** | **RAG (Retrieval-Augmented Generation)**: 신뢰할 수 있는 지식 베이스(Graphify, LLMWiki)에서 정보를 검색하여 LLM의 답변을 보강하고, **검증(Verification) 레이어**를 통해 LLM 출력의 사실 여부를 교차 검증합니다. | Graphify, LLMWiki, CrewAI (검증 에이전트) |
| **불안정한 출력 형식 및 일관성 부족** | **구조화된 출력 강제**: Pydantic 등의 라이브러리를 활용하여 LLM의 출력을 특정 JSON 스키마나 Markdown 형식으로 강제하고, **템플릿 기반 생성**을 통해 일관성을 유지합니다. | LLMWiki, CrewAI (출력 포맷팅 에이전트) |
| **긴 문맥 처리의 한계 (Context Window)** | **압축된 메모리 및 지식 그래프**: Mem0를 통해 핵심 정보만 압축하여 LLM에 전달하고, Graphify 지식 그래프를 통해 필요한 정보만 선별적으로 제공하여 문맥 길이를 최적화합니다. | Mem0, Graphify |
| **실시간성 및 최신 정보 반영 부족** | **지속적인 지식 업데이트 파이프라인**: OpenClaw를 통해 실시간으로 데이터를 수집하고, LLMWiki 및 Graphify를 통해 지식 베이스를 지속적으로 업데이트합니다. | OpenClaw, LLMWiki, Graphify |
| **비용 효율성** | **모듈별 LLM 최적화**: 각 모듈의 역할에 따라 최적화된 소형 LLM (예: 요약 전용 LLM, 관계 추출 전용 LLM)을 사용하거나, 필요에 따라 대형 LLM을 호출하여 비용을 절감합니다. | 모든 LLM 활용 모듈 |

## 2. 모듈형 시스템 아키텍처 상세 설계

시스템은 크게 4가지 핵심 모듈로 구성되며, 각 모듈은 독립적인 API를 통해 상호작용합니다. 이를 통해 특정 모듈의 변경이 다른 모듈에 미치는 영향을 최소화하고, 개별 모듈의 개발 및 배포를 용이하게 합니다.

### 2.1. 데이터 수집 모듈 (Data Ingestion Module)
*   **역할**: 다양한 외부 소스(파일, 이메일, 웹사이트, 메신저)로부터 원시 데이터를 수집하고 정규화하여 다음 모듈로 전달합니다.
*   **주요 기능**: 파일 파싱, 이메일/메신저 API 연동, 웹 크롤링, 데이터 정규화.
*   **핵심 오픈소스**: **OpenClaw** (멀티 채널 게이트웨이), `BeautifulSoup` (웹 크롤링), `email` 라이브러리 (이메일 파싱).
*   **LLM 활용**: 초기 데이터 분류 및 필터링 (예: 스팸 제거, 중요도 분류).

### 2.2. 지식 처리 모듈 (Knowledge Processing Module)
*   **역할**: 수집된 원시 데이터를 가공하여 구조화된 지식으로 변환하고, 지식 그래프를 구축합니다.
*   **주요 기능**: 텍스트 요약, 키워드/개체 추출, 관계 추출, 지식 그래프 저장 및 관리.
*   **핵심 오픈소스**: **LLMWiki** (Markdown 구조화), **Graphify** (지식 그래프 구축 및 관리), `spaCy` 또는 `NLTK` (NLP 전처리).
*   **LLM 활용**: LLMWiki 패턴에 따른 요약 및 재구성, Graphify를 위한 개체 및 관계 추출.

### 2.3. 지능형 메모리 및 추론 모듈 (Intelligent Memory & Reasoning Module)
*   **역할**: 사용자의 맥락과 선호도를 기억하고, 지식 그래프를 기반으로 추론하여 실행 가능한 통찰을 도출합니다.
*   **주요 기능**: 사용자 프로필 관리, 장기/단기 기억 관리, 추론 엔진, 우선순위 결정.
*   **핵심 오픈소스**: **Mem0** (개인화된 메모리 관리), `LangChain` 또는 `LlamaIndex` (추론 및 RAG 프레임워크).
*   **LLM 활용**: 사용자 선호도 학습, 지식 그래프 질의 응답, 복잡한 추론 및 의사결정 지원.

### 2.4. 실행 및 알림 모듈 (Action & Notification Module)
*   **역할**: 추론 모듈에서 도출된 할 일 및 지시사항을 사용자에게 전달하고, 필요한 경우 외부 시스템(메신저, 캘린더)에 액션을 수행합니다.
*   **주요 기능**: 할 일 목록 생성, 알림 전송, 외부 시스템 연동 (캘린더 이벤트 생성, 메시지 발송).
*   **핵심 오픈소스**: **CrewAI** (멀티 에이전트 워크플로우), **OpenClaw** (알림 게이트웨이 재활용), **Obsidian** (로컬 지식 허브).
*   **LLM 활용**: 최종 할 일 목록의 자연어 생성, 알림 메시지 작성, 팀원별 맞춤형 지시사항 생성.

## 3. 모듈 간 유기적 연동 및 데이터 흐름

각 모듈은 RESTful API 또는 메시지 큐(예: RabbitMQ, Kafka)를 통해 비동기적으로 통신합니다. 이를 통해 모듈 간의 의존성을 낮추고, 시스템의 전반적인 처리량과 안정성을 향상시킵니다.



## 4. 유지보수 및 확장성

*   **독립적인 배포**: 각 모듈은 독립적인 서비스로 배포될 수 있어, 특정 모듈의 업데이트나 스케일링이 용이합니다.
*   **기술 스택 유연성**: 각 모듈은 최적의 기술 스택을 선택할 수 있으며, LLM 모델 교체 시에도 해당 모듈만 수정하면 됩니다.
*   **API 기반 연동**: 표준화된 API 인터페이스를 통해 모듈 간의 결합도를 낮추고, 새로운 모듈 추가나 기존 모듈 교체가 쉽습니다.
*   **테스트 용이성**: 각 모듈을 독립적으로 테스트할 수 있어, 버그 발견 및 수정이 효율적입니다.

이 아키텍처는 LLM의 강력한 기능을 활용하면서도 그 한계를 보완하고, 미래의 변화에 유연하게 대응할 수 있는 견고한 기반을 제공할 것입니다.
